# Amazon S3 Pre-signed URL Hands-on Lab

Bài thực hành này hướng dẫn bạn các bước chi tiết để cấu hình và kiểm nghiệm tính năng **Pre-signed URL** (Đường dẫn ký trước) trên Amazon S3 nhằm cấp quyền truy cập tạm thời cho một đối tượng riêng tư.

---

## Các bước thực hiện chi tiết

### Bước 1: Kiểm tra cấu hình kết nối AWS CLI
Trước khi thực hiện ký tạo URL từ dòng lệnh, bạn cần đảm bảo máy tính đã được cài đặt và cấu hình thông tin xác thực AWS CLI chính xác:

* Mở Terminal (Command Prompt hoặc PowerShell).
* Chạy lệnh sau để kiểm tra danh tính tài khoản IAM hiện tại đang kết nối:
  ```bash
  aws sts get-caller-identity
  ```
  * Xác nhận kết quả trả về hiển thị đúng `UserId`, `Account` và `Arn` của người dùng IAM của bạn (ví dụ: `developer-01`).
* Tiếp tục chạy lệnh hiển thị danh sách các S3 Buckets đang quản lý để kiểm nghiệm kết nối:
  ```bash
  aws s3 ls
  ```
  * Xác nhận trong danh sách hiển thị có chứa bucket bạn đang thực hành (ví dụ: `h1eudayne`).

![Kiểm tra kết nối AWS CLI](../../../../images/aws/s3_presigned_url_lab_step1_cli.png)

### Bước 2: Upload file lên S3 và kiểm tra chặn truy cập công khai (Block Public Access)
Để kiểm tra tính bảo mật của Pre-signed URL, chúng ta cần chắc chắn rằng đối tượng trên S3 đang ở chế độ riêng tư hoàn toàn (mặc định bị chặn truy cập công khai):

* **B1 (Upload đối tượng)**: Upload một tệp tin (ví dụ: hình ảnh `Ảnh chụp màn hình...` hoặc tệp tài liệu) lên S3 Bucket của bạn.
* **B2 (Kiểm tra cấu hình chặn truy cập công khai)**:
  * Truy cập vào S3 Bucket của bạn trên AWS Console.
  * Chọn tab **Permissions** (Quyền truy cập).
  * Tìm đến mục **Block public access (bucket settings)** và đảm bảo tùy chọn **Block *all* public access** đang ở trạng thái **On** (Đang bật).
  
  ![Kiểm tra Block Public Access](../../../../images/aws/s3_presigned_url_lab_step2_block_public.png)

  * *Lưu ý*: Nếu tùy chọn này chưa bật, nhấp vào nút **Edit** ở bên phải, tích chọn **Block all public access**, bấm **Save changes** để hoàn tất kích hoạt.
  
  ![Cập nhật Block Public Access](../../../../images/aws/s3_presigned_url_lab_step2_edit_block.png)

* **B3 (Xác nhận trạng thái riêng tư - AccessDenied)**:
  * Click trực tiếp vào đối tượng bạn vừa upload, lấy đường dẫn URL công khai của đối tượng (dạng: `https://<bucket-name>.s3.<region>.amazonaws.com/<key>`).
  * Mở trình duyệt web ẩn danh và dán đường dẫn này để truy cập.
  * Xác nhận trình duyệt trả về lỗi cấu trúc XML thông báo **`AccessDenied`** (Không cho phép truy cập). Điều này chứng minh tệp tin đang được bảo mật an toàn và không thể xem trực tiếp qua internet.

![Trình duyệt hiển thị AccessDenied](../../../../images/aws/s3_presigned_url_lab_step2_access_denied.png)

### Bước 3: Phát hành và kiểm nghiệm S3 Pre-signed URL từ AWS CLI
Trong bước này, chúng ta sẽ thực hiện tạo một đường dẫn ký trước Pre-signed URL bằng dòng lệnh CLI có hiệu lực giới hạn trong vòng **60 giây** và kiểm tra hành vi truy cập.

* **B1 (Lấy S3 URI của tệp tin)**:
  * Trên giao diện AWS Console, nhấp vào tệp tin đã upload.
  * Trong tab **Properties** (Thuộc tính) của tệp tin, bấm vào nút **Copy S3 URI** ở góc trên cùng bên phải.
  * Định dạng của S3 URI sẽ có dạng: `s3://<bucket-name>/<prefix>/<filename>`.

![Sao chép S3 URI](../../../../images/aws/s3_presigned_url_lab_step3_copy_uri.png)

* **B2 (Tạo Pre-signed URL từ AWS CLI)**:
  * Sử dụng lệnh `aws s3 presign` kèm theo S3 URI vừa sao chép và tùy chọn `--expires-in 60` (để thiết lập thời hạn hết hiệu lực sau 60 giây).
  * Do bucket của bạn nằm ở region `us-east-1`, hãy chỉ định thêm tùy chọn `--region us-east-1` nếu cần.
  * Cú pháp lệnh:
    ```bash
    aws s3 presign "s3://h1eudayne/test/anh chup man hinh 2025-07-19 142928.png" --region us-east-1 --expires-in 60
    ```
  * AWS CLI sẽ sinh ra và trả về một đường dẫn URL dài tích hợp sẵn chữ ký số xác thực.

![Tạo Pre-signed URL bằng AWS CLI](../../../../images/aws/s3_presigned_url_lab_step3_presign.png)

* **B3 (Truy cập thử nghiệm trong thời gian hiệu lực)**:
  * Sao chép toàn bộ đường dẫn URL được sinh ra ở trên và dán vào trình duyệt web.
  * Xác nhận tệp tin (ví dụ ở đây là ảnh chụp màn hình mockup trang Login) được tải hoặc hiển thị thành công bình thường.

![Truy cập URL thành công](../../../../images/aws/s3_presigned_url_lab_step3_success.png)

* **B4 (Kiểm tra hành vi sau khi URL hết hạn)**:
  * Đợi hết thời gian 60 giây, sau đó tải lại (refresh) trang trình duyệt đó.
  * Xác nhận trình duyệt trả về lỗi cấu trúc XML **`AccessDenied`** với thông điệp chi tiết **`<Message>Request has expired</Message>`** (Yêu cầu đã hết hạn). Điều này chứng minh Pre-signed URL đã tự động vô hiệu hóa đúng thời hạn thiết lập.

![URL hết hạn báo lỗi Request has expired](../../../../images/aws/s3_presigned_url_lab_step3_expired.png)

### * Lưu ý quan trọng trong thực tế (Production Best Practices):
Trong các dự án thực tế, quản trị viên và lập trình viên **không sử dụng AWS CLI** để sinh Pre-signed URL thủ công cho người dùng.
* Thay vào đó, chúng ta sẽ xây dựng một **Backend Server** (ví dụ viết bằng Node.js, Python, Java...).
* Backend Server này chạy trên một tài nguyên AWS (như EC2, ECS, Lambda) được gán một **IAM Instance Profile / IAM Role** có đầy đủ quyền đọc/ghi trên S3 (`s3:GetObject` / `s3:PutObject`).
* Khi người dùng gửi request muốn tải/tải lên file, Backend Server sẽ chạy code SDK (như Python `boto3`, Node.js `@aws-sdk/s3-request-presigner`...) để tự động sinh ra Pre-signed URL và trả về cho ứng dụng Client truy cập trực tiếp.
