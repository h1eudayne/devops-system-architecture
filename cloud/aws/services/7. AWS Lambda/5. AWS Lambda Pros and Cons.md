# 5. AWS Lambda Pros and Cons (Ưu điểm và nhược điểm)

Trước khi quyết định đưa AWS Lambda vào kiến trúc ứng dụng của doanh nghiệp, bạn cần phân tích kỹ lưỡng các ưu điểm vượt trội và nhược điểm cố hữu của mô hình tính toán serverless này.

---

## I. Ưu điểm (Pros)

### 1. Không cần quản lý máy chủ (No Server Management)
* Không cần cài đặt hệ điều hành, cấu hình firewall, vá lỗi bảo mật phần cứng, hay bảo trì hạ tầng.
* Các kỹ sư có thể tập trung hoàn toàn vào việc viết code và phát triển logic nghiệp vụ (business logic).

### 2. Tự động co giãn linh hoạt (Continuous Scaling)
* Lambda tự động co giãn theo chiều ngang (scale-out) khi số lượng request tăng đột biến, và thu hẹp lại khi tải giảm mà không cần can thiệp thủ công hay viết script auto-scaling phức tạp.
* Chạy song song lên tới hàng ngàn hàm đồng thời.

### 3. Tối ưu hóa chi phí tối đa (Cost Efficiency)
* idle cost = 0. Bạn hoàn toàn không mất đồng nào nếu hàm không hoạt động.
* Thời gian thanh toán tính chính xác theo mili-giây thời gian code chạy thực tế.

### 4. Triển khai nhanh chóng (Fast Time-to-Market)
* Việc cập nhật code diễn ra chỉ trong vài giây.
* Dễ dàng tích hợp với các công cụ CI/CD để tự động hóa việc deploy.

---

## II. Nhược điểm (Cons)

### 1. Trễ khởi động nguội (Cold Starts)
* **Vấn đề**: Khi một hàm Lambda đã lâu không được gọi hoặc khi hệ thống scale-out mở thêm container mới, AWS sẽ mất một khoảng thời gian (từ vài trăm mili-giây đến vài giây) để khởi tạo môi trường chạy và tải gói code của bạn.
* **Hệ quả**: Gây ra độ trễ (latency) cho request đầu tiên, điều này có thể không phù hợp với các ứng dụng nhạy cảm về thời gian phản hồi thời gian thực (như các API giao dịch tài chính).

### 2. Giới hạn tài nguyên và thời gian thực thi (Execution Limits)
* **Thời gian tối đa**: Giới hạn timeout cứng tối đa là **15 phút**. Không thể dùng Lambda cho các tác vụ xử lý dữ liệu nặng chạy liên tục nhiều giờ.
* **Vùng nhớ tạm `/tmp`**: Tối đa **10 GB**, giới hạn khả năng xử lý các file dữ liệu quá lớn trên đĩa cứng cục bộ.

### 3. Khó khăn trong việc Debug và Giám sát (Debugging & Monitoring)
* Do chạy trong môi trường phân tán hoàn toàn serverless, bạn không thể SSH trực tiếp vào container để debug khi có lỗi xảy ra.
* Việc tracking luồng dữ liệu (trace) giữa nhiều dịch vụ serverless đòi hỏi phải sử dụng các công cụ bổ trợ phức tạp như AWS X-Ray.

### 4. Phụ thuộc nhà cung cấp (Vendor Lock-in)
* Mã nguồn viết cho AWS Lambda kết hợp sâu với các dịch vụ khác của AWS (như DynamoDB, S3) sẽ rất khó khăn nếu muốn chuyển đổi sang các đám mây khác (như Google Cloud Functions hay Azure Functions).
