# Amazon S3 Bucket Policy & Access Control List (ACL)

## I. Tổng quan về bảo mật mức tài nguyên (Resource-level Security)

Bên cạnh việc sử dụng **IAM Policy** (quản lý phân quyền dựa trên danh tính người dùng - Identity-based) mà bạn đã học ở phần trước, Amazon S3 còn hỗ trợ **Resource-level Policy** (chính sách đính kèm trực tiếp vào tài nguyên) để kiểm soát quyền truy cập chặt chẽ hơn. 

Hai cơ chế quản lý bảo mật mức tài nguyên chính của S3 là:
1. **S3 Bucket Policy**: Chính sách phân quyền dạng JSON đính kèm trực tiếp vào Bucket.
2. **Access Control List (ACL)**: Danh sách kiểm soát truy cập ở mức Bucket hoặc từng Object (cơ chế truyền thống).

---

## II. S3 Bucket Policy là gì?

**S3 Bucket Policy** là tài liệu chính sách bảo mật viết dưới dạng mã **JSON**, được gán trực tiếp lên một Bucket cụ thể để kiểm soát quyền truy cập của người dùng hoặc các dịch vụ khác đối với bucket đó và toàn bộ tài nguyên con bên trong nó (thư mục, tệp tin).

*   **Bản chất hoạt động**: Hoạt động hoàn toàn tương tự như một IAM Policy thông thường, nhưng thay vì đính kèm vào User hay Role, nó được đính kèm trực tiếp vào S3 Bucket.
*   **Phạm vi áp dụng**: Chỉ có hiệu lực trong phạm vi của chính Bucket đó và các đối tượng (Objects) nằm bên trong.
*   **Cơ chế hoạt động**: Định nghĩa rõ ràng việc **Cho phép (Allow)** hoặc **Từ chối (Deny)** các hành động cụ thể (Actions) của đối tượng truy cập (Principals) lên Bucket và các đối tượng con.
*   **Trường hợp sử dụng**:
    *   Áp dụng cho những Bucket yêu cầu mức độ bảo mật cao, cần cấu hình giới hạn truy cập một cách chặt chẽ.
    *   Cần cấp quyền truy cập chéo tài khoản (Cross-account access) cho người dùng thuộc tài khoản AWS khác.
    *   Bắt buộc mã hóa dữ liệu khi truyền tải (chỉ cho phép truy cập qua HTTPS).
    *   Chỉ cho phép truy cập từ một dải IP cụ thể (IP Whitelisting).

## III. Cấu trúc của một S3 Bucket Policy

Một tệp chính sách S3 Bucket Policy dạng JSON cơ bản bao gồm các thành phần cốt lõi sau:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "ExampleStatementID",
      "Effect": "Allow",
      "Principal": "*",
      "Action": "s3:GetObject",
      "Resource": "arn:aws:s3:::ten-bucket-cua-ban/*",
      "Condition": {
        "IpAddress": {
          "aws:SourceIp": "203.0.113.0/24"
        }
      }
    }
  ]
}
```

### Chi tiết các thành phần:
*   **`Version`**: Phiên bản ngôn ngữ chính sách (thường cố định là `"2012-10-17"`).
*   **`Sid` (Statement ID)**: Định danh tùy chọn cho từng Statement bên trong policy.
*   **`Effect`**: Chỉ định hành động là cho phép (`Allow`) hoặc cấm (`Deny`).
*   **`Principal`**: Đối tượng thực thi truy cập (có thể là một User cụ thể, một tài khoản AWS khác, hoặc dấu `"*"` đại diện cho tất cả mọi người).
*   **`Action`**: Các API hành động của S3 muốn áp dụng (ví dụ: `s3:GetObject` để đọc file, `s3:PutObject` để ghi file, `s3:ListBucket` để xem danh sách file).
*   **`Resource`**: Định danh tài nguyên AWS (ARN) của Bucket và đối tượng chịu ảnh hưởng (ví dụ: `arn:aws:s3:::my-bucket` cho mức bucket, hoặc `arn:aws:s3:::my-bucket/*` cho tất cả đối tượng bên trong bucket).
*   **`Condition`**: Điều kiện đi kèm để chính sách có hiệu lực (ví dụ: yêu cầu địa chỉ IP nguồn, yêu cầu kết nối HTTPS bảo mật).

## IV. So sánh IAM Policy và S3 Bucket Policy

| Tiêu chí | IAM Policy | S3 Bucket Policy |
| :--- | :--- | :--- |
| **Loại chính sách** | Identity-based (Gắn với danh tính) | Resource-based (Gắn với tài nguyên) |
| **Đối tượng đính kèm** | IAM User, Group, hoặc IAM Role | Chỉ đính kèm trực tiếp vào S3 Bucket |
| **Phạm vi quản lý** | Có thể quản lý quyền truy cập của User đó trên nhiều dịch vụ AWS khác nhau (EC2, S3, RDS...). | Chỉ quản lý quyền truy cập của mọi đối tượng đối với duy nhất Bucket được đính kèm. |
| **Xác định Principal** | Không cần khai báo (vì đã mặc định là User/Role được gán). | Bắt buộc phải khai báo trường `Principal` trong JSON để chỉ rõ ai được phép/cấm truy cập. |
| **Sử dụng tốt nhất khi** | Bạn muốn quản lý quyền hạn của từng người dùng cụ thể trong hệ thống ("Nhân viên A được làm gì trên AWS?"). | Bạn muốn quản lý quyền truy cập vào một vùng lưu trữ cụ thể ("Ai được phép đọc/ghi dữ liệu trong Bucket nhạy cảm này?"). |

## V. Access Control List (ACL) trong S3

**S3 Access Control List (ACL)** là cơ chế kiểm soát truy cập truyền thống (legacy) có từ thời kỳ đầu của AWS. ACL cho phép bạn phân quyền ở mức chi tiết đến từng đối tượng (Object-level) hoặc mức Bucket cho các tài khoản AWS khác hoặc các nhóm định sẵn của hệ thống.

*   **Hạn chế của ACL**: ACL không sử dụng cú pháp JSON linh hoạt, khó quản lý ở quy mô lớn, và dễ dẫn đến rủi ro cấu hình sai gây lộ lọt dữ liệu.
*   **Khuyến nghị hiện tại**: AWS khuyến nghị tắt tính năng ACL (**ACLs disabled**) cho tất cả các Bucket mới tạo và sử dụng hoàn toàn **S3 Bucket Policy** kết hợp với **IAM Policy** để quản lý phân quyền tập trung và bảo mật hơn.

## VI. Các ví dụ thực tế về S3 Bucket Policy

### 1. Bắt buộc truy cập qua kết nối bảo mật HTTPS (Enforce HTTPS)
Chính sách này sẽ từ chối tất cả các hành động trên bucket nếu kết nối không được mã hóa bảo mật qua HTTPS (SSL/TLS).

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "EnforceHTTPSOnly",
      "Effect": "Deny",
      "Principal": "*",
      "Action": "s3:*",
      "Resource": [
        "arn:aws:s3:::my-secure-bucket",
        "arn:aws:s3:::my-secure-bucket/*"
      ],
      "Condition": {
        "Bool": {
          "aws:SecureTransport": "false"
        }
      }
    }
  ]
}
```

### 2. Giới hạn chỉ cho phép truy cập từ địa chỉ IP của công ty (IP Restriction)
Chính sách này chỉ cho phép đọc dữ liệu trong bucket từ dải IP nội bộ được định sẵn của doanh nghiệp.

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowIPRestrictionOnly",
      "Effect": "Allow",
      "Principal": "*",
      "Action": "s3:GetObject",
      "Resource": "arn:aws:s3:::my-corporate-bucket/*",
      "Condition": {
        "IpAddress": {
          "aws:SourceIp": "192.168.1.0/24"
        }
      }
    }
  ]
}
```

### 3. Cho phép một IAM User cụ thể có quyền đọc dữ liệu (Allow Specific User Read)
Chính sách này cho phép người dùng IAM `user1` thuộc tài khoản AWS `614073292094` có quyền đọc đối tượng (`s3:GetObject`) từ S3 Bucket tên là `abc-bucket-sample-001`.

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "Allow_user1_read",
      "Effect": "Allow",
      "Principal": {
        "AWS": [
          "arn:aws:iam::614073292094:user/user1"
        ]
      },
      "Action": [
        "s3:GetObject"
      ],
      "Resource": "arn:aws:s3:::abc-bucket-sample-001/*"
    }
  ]
}
```
